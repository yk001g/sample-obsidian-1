# サービス開発全体の流れ整理

## 概要
サービス開発における機能追加の全体フローと、各フェーズで使用するプロンプトの整理

## タグ
#development-workflow #process #cursor #project-management

## 内容

### 機能追加の全体フロー

1. **機能追加会議**
2. **追加機能投票ランキング作成**
3. **ランキング1位の機能を決定**
   - 参考プロンプト「天才アイデア会議ランキング表」
   - アイデア100個考えて水平思考で、そのあと、開発チーム側のニーズや意見と、ユーザー側のニーズがわかる一般人審査員キャラ1000名が投票したランキング表を作成

4. **1位の機能を仕様書追加・設計・実装・レビュー依頼**
   - 要件仕様書追加: `docs/requirements`に今回の分の仕様を要件定義のmdファイルに追記
   - 設計: `docs/specs`に今回の分の実装の設計をmdファイルで追記、`docs/tasks`に「タスクindex_タスクタイトル」のフォルダを作成して今回のタスク内容を書いたmdファイルを作成、同じフォルダに今回の実装手順を書く
   - 実装: 今回のタスク通りに実装
   - 単体テスト: `docs/tasks`の今回の「タスクindex番号n_タスクタイトル」のフォルダの中に、「単体テストindex番号m_タスクタイトル」フォルダを作成して、今回の実装が合っているかどうか単体レビューテストケースが書かれたmdファイルを作成、単体レビューに問題ないか実際にjestでチェック、問題あれば問題が消えるまでテストコードと実装コードの改善を繰り返す
   - 結合テスト: 単体レビューが問題なければ、`docs/requirements`と`docs/specs`と今回のタスク内容を見比べて、「タスクindex番号n_タスクタイトル」のフォルダの中に、「結合テストindex番号m_タスクタイトル」フォルダを作成して、今回の実装が合っているかどうか結合テストレビューテストケースが書かれたmdファイルを作成、結合テストレビューに問題ないか実際にplaywrightでチェック、問題あれば問題が消えるまでテストコードと実装コードの改善を繰り返す（mongodbは.envにある環境をそのまま使用）

5. **npm run build通す**
6. **buildが通ったら、git add . してgit commit -m "わかりやすい日本語の説明"してgit push origin main**

### ワークフロー管理
- 以上の流れをシェルコマンドでcursorへ指示して、時系列で、今どのステップにいるかを理解した上で、今のcursorへの指示を選択して続ける
- 最新の作業ログを残す
- 私のやりたいことを`docs/needs`へ整理
