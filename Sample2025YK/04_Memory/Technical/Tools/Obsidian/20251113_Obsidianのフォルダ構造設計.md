# Obsidianのフォルダ構造設計

## 概要
Obsidianのフォルダ構造は、知識を整理するための重要な要素です。適切なフォルダ構造を設計することで、ノートの管理が容易になり、知識の体系化が進みます。

## 詳細

### フォルダ構造の基本原則

#### 原則1: シンプルに保つ
```
✅ 良い例:
00_Memo/
01_Inbox/
02_Daily/
04_Memory/
05_Output/

❌ 悪い例:
00_Memo/
00_Memo/2025/
00_Memo/2025/11/
00_Memo/2025/11/13/
（深すぎる階層）
```

#### 原則2: 明確な目的
```
各フォルダには明確な目的がある:

00_Memo: 何でも放り込む場所
01_Inbox: Cursor処理済み
02_Daily: デイリーノート
04_Memory: 長期記憶
05_Output: アウトプット管理
```

#### 原則3: フォルダとタグの使い分け
```
フォルダ:
- プロジェクト単位の分類
- 期間単位の分類
- 明確な階層構造

タグ:
- 横断的な分類
- 複数のカテゴリーに属する場合
```

## 基本的な使い方

### 推奨フォルダ構造

#### PARA法ベースの構造
```
SecondBrain/
├─ 00_Memo/          # 何でも放り込む
├─ 01_Inbox/         # 処理待ち
├─ 02_Daily/         # デイリーノート
├─ 03_Input/         # 短期記憶
├─ 04_Memory/        # 長期記憶
├─ 05_Output/        # アウトプット
│  ├─ Projects/      # プロジェクト
│  └─ Areas/         # 継続的エリア
├─ 06_Templates/     # テンプレート
├─ 07_System/        # システム設定
└─ 99_Archive/        # アーカイブ
```

#### カテゴリーベースの構造
```
SecondBrain/
├─ Daily/            # デイリーノート
├─ Projects/         # プロジェクト
├─ Areas/            # 継続的エリア
├─ Resources/        # リソース
│  ├─ AI/
│  ├─ Business/
│  └─ Technical/
└─ Archive/          # アーカイブ
```

## 高度な使い方

### フォルダ命名規則

#### 推奨命名規則
```
✅ 良い例:
- 00_Memo（番号で順序付け）
- 04_Memory（明確な目的）
- Projects/@Active（ステータス管理）

❌ 悪い例:
- memo（目的が不明確）
- フォルダ1（意味不明）
- 新しいフォルダ（一時的な名前）
```

#### 命名パターン
```
1. 番号付きフォルダ
   00_Memo, 01_Inbox, 02_Daily

2. ステータス管理
   Projects/@Active, Projects/@Completed

3. 日付フォルダ
   2025/2025-11/2025-11-13
```

### フォルダの階層設計

#### 浅い階層（推奨）
```
✅ 良い例:
04_Memory/AI/Tools/Cursor/

深さ: 3階層
メリット: ナビゲーションが容易
```

#### 深い階層（避ける）
```
❌ 悪い例:
04_Memory/AI/Tools/Cursor/Tips/Basic/2025/11/

深さ: 7階層
デメリット: ナビゲーションが困難
```

### フォルダとリンクの組み合わせ

#### ハイブリッド戦略
```
フォルダ:
- 大まかな分類（AI、Business、Technical）

リンク:
- 細かい分類（概念間の関係性）

例:
04_Memory/AI/Tools/Cursor.md
→ [[AI Tools]] [[Cursor]] などのリンクで繋ぐ
```

## メリット・デメリット

### ✅ メリット
1. **整理**: ノートを体系的に整理
2. **ナビゲーション**: フォルダでノートを探しやすい
3. **管理**: プロジェクト単位で管理可能
4. **バックアップ**: フォルダ単位でバックアップ可能

### ❌ デメリット
1. **柔軟性**: フォルダは1つのノートを1つの場所にしか置けない
2. **階層の深さ**: 深すぎるとナビゲーションが困難
3. **メンテナンス**: フォルダ構造の変更が大変

## 使用例

### シナリオ1: 個人の知識管理
```
構造:
00_Memo/          # 一時的なメモ
01_Inbox/         # 処理待ち
02_Daily/         # デイリーノート
04_Memory/        # 長期記憶
  ├─ AI/
  ├─ Business/
  └─ Technical/
05_Output/        # アウトプット
  ├─ Projects/
  └─ Areas/

→ シンプルで管理しやすい構造
```

### シナリオ2: チームでの使用
```
構造:
Shared/           # 共有フォルダ
  ├─ Projects/
  ├─ Resources/
  └─ Templates/
Personal/         # 個人フォルダ
  ├─ Daily/
  └─ Notes/

→ 共有と個人を分離
```

## よくある問題と解決策

### 問題1: フォルダが増えすぎる
**解決策**:
- フォルダ数を制限（10-15個程度）
- タグとリンクを活用
- 定期的にフォルダを見直す

### 問題2: どのフォルダに保存すべきか迷う
**解決策**:
- 明確なルールを設定
- デフォルトフォルダを設定
- 迷ったら01_Inboxに保存

### 問題3: フォルダ構造を変更したい
**解決策**:
- 段階的に変更（一度にすべてを変更しない）
- バックアップを取る
- リンクの更新を確認

## ベストプラクティス

1. **シンプルに**: 深すぎる階層は避ける
2. **明確な目的**: 各フォルダに明確な目的がある
3. **番号付け**: 順序が重要な場合は番号を付ける
4. **タグと組み合わせ**: フォルダ + タグのハイブリッド戦略
5. **定期的に見直す**: フォルダ構造を定期的に見直す
